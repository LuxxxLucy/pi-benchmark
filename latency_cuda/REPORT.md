# pi-benchmark — Lightweight-classifier latency on RTX 3090 (CUDA + CPU)

**Updated:** 2026-04-20
**Host:** Linux-6.8, NVIDIA RTX 3090 24GB, torch 2.6.0+cu124
**Scope:** batch=1 forward-pass P50/P95 latency across 8 token lengths (32 / 64 / 128 / 256 / 512 / 1024 / 2048 / 4096), same-machine CUDA (bfloat16) and CPU (float32) passes, 24 encoder candidates (7 PI-trained + 17 architecture baselines). 22 measured, 2 failed tokenizer load (bert-tiny, bert-mini: WordPiece-only repos that trip the slow→fast tokenizer converter even with `use_fast=False`). Three encoders (modernbert-base, ettin-encoder-150m, gte-base-en-v1.5) run the full 32–4096 length sweep; the rest cap at max_position_embeddings=512.
**Source:** `src/bench.py` + `src/models.py`; raw stdout in `results/3090_run_2026-04-20.log`; partial reconstructed JSON in `results/latency_{cuda,cpu}.json` (`source: "log_reconstruct"` — p50/p95/mean/throughput only, regenerated by `scripts/rebuild_json_from_log.py`).

## TL;DR

1. **On a 3090, every 11M–184M encoder in the non-DeBERTa, non-MobileBERT, non-ALBERT set runs 1.4–17 ms P50 at batch=1 across 32–512 tokens. If A6 deploys on GPU, latency is not a constraint on backbone choice below ~150M.**
2. **Fewer layers beats fewer params at batch=1 on GPU.** bge-micro-v2 (17.4M, 3-layer / 384-hidden) runs 1.4–2.1 ms P50 across 32–512 — *faster* than testsavantai-tiny (4.4M, 6-layer / 384-hidden) at most lengths. Kernel dispatch count dominates parameter count in the sub-30M regime.
3. **GTE-base-en-v1.5 is the long-context winner at 137M.** On GPU it holds 5–6 ms P50 through 512 tokens, 8.5 ms @ 1024, 17.4 ms @ 2048, 39 ms @ 4096 — 1.7× faster than ModernBERT / Ettin at 4096 (both 67.8 ms). A standard transformer body with RoPE beats ModernBERT's alternating local/global attention on this hardware at batch=1.
4. **ModernBERT @ 4096 tokens fits A6's 200 ms CPU budget on GPU comfortably** — 67.8 ms P50 / 114.7 ms P95. At 2048 tokens it is 29.6 ms P50. Both Ettin and ModernBERT land within measurement noise of each other, consistent with Ettin being a retraining of ModernBERT.
5. **MobileBERT and ALBERT are GPU anti-patterns.** MobileBERT (24.6M) runs 17.7–18.5 ms P50 regardless of length — ~7× slower than bge-small-en-v1.5 (33.4M, 5 ms) despite fewer params. ALBERT-base-v2 (11.7M, shared across 12 layers) runs 5.8–7.0 ms — 3× slower than bert-L4-H256 (11.2M) at equivalent params. Parameter sharing and deep-narrow inverted-bottleneck designs target mobile CPU, not server GPU.
6. **DeBERTa-v3 CUDA tax persists at 4–6× DistilBERT.** deberta-v3-xsmall (70.8M) runs 12.5 ms vs distilbert-base (67.0M) at 2.4 ms @ 512 — *5.1× slower at matched params*. The PI-trained 184M DeBERTa variants run 13.5–14.1 ms. Disentangled-attention dispatch, not parameter count, drives the gap. **Avoid DeBERTa-v3 on GPU if cost per inference matters.**
7. **CPU findings replicate the M3 baseline.** 66M DistilBERT (fmops) runs 83 ms P50 @ 512, still 2.4× under A6's 200 ms budget. 184M DeBERTa blows the budget at 512 (395–401 ms) and at 256 (140–142 ms, narrow). ALBERT on CPU is catastrophic (194 ms @ 512, 12.7× worse than bge-small); parameter sharing buys memory, not compute. Long-context encoders die on CPU past 1024 tokens (modernbert 563 ms; all three >3 s at 4096).

## 1. Setup

**Method.** Per `(model, length)`: 10 warmup forwards, 50 measured forwards, `torch.cuda.synchronize()` around each, `torch.no_grad()`, gc disabled inside the measure loop, batch_size=1. bf16 on CUDA, fp32 on CPU. Input is `FILL_TEXT` looped and truncated to exact target length (see `../bench_common.py`). Lengths > `max_position_embeddings` are skipped per model (only modernbert-base, ettin-encoder-150m, gte-base-en-v1.5 admit 1024/2048/4096).

**Models.** 24 encoders in two groups (`src/models.py`):
- **PI-trained (7)** — real prompt-injection classifiers with task-fine-tuned heads: `testsavantai-{tiny,small,medium,base}`, `fmops-distilbert`, `protectai-deberta-v2`, `deepset-deberta-injection`.
- **Arch-baseline (17)** — open backbones with random classifier heads. Latency-only. Covers the bert-tiny / distilbert / minilm / deberta families plus modern long-context entrants (`modernbert-base`, `ettin-encoder-150m`, `gte-base-en-v1.5`) and design-space probes for parameter sharing (`albert-base-v2`), factorized embedding (`electra-small`), and mobile inverted-bottleneck (`mobilebert`).

**Failures.** `prajjwal1/bert-{tiny,mini}` fail tokenizer load on this transformers version: the repos ship only `vocab.txt`, the slow→fast converter imports `sentencepiece` / `tiktoken` regardless, and `use_fast=False` still routes through `PreTrainedTokenizerFast.__init__`. Fixed by swapping the 28M cell to `google/bert_uncased_L-4_H-512_A-8` (ships `tokenizer.json`), which covers the same architectural slot.

**What this report does not cover.** Accuracy (parent [REPORT.md](../../REPORT.md)), quantization (INT8 / AWQ not measured), batched throughput (batch > 1), production engines (eager PyTorch only; no torch.compile, TensorRT, ONNX-Runtime).

### 2.1 Long-context — only three models admit > 512 tokens

| modernbert-base | 149.6M | 8192 | 15.64 / 32.19 | 29.58 / 54.85 | 67.76 / 114.69 |

**GTE-base holds ~40 ms P50 at 4096 tokens** — 1.7× faster than ModernBERT / Ettin despite comparable param count. The attention pattern matters more than the embedding dimension at these lengths: GTE is a standard transformer with RoPE, one matmul path per head; ModernBERT alternates local sliding-window and global attention, which dispatches a separate global-attention kernel every few layers. At 4096 tokens both architectures fit comfortably under A6's 200 ms gate on GPU.

**Ettin is a ModernBERT retraining, confirmed by latency.** The two profiles are bit-equivalent within measurement noise across all lengths. No new optimizer in Ettin; same compute shape.

### 2.2 Observations

**GPU latency is near-flat across token length below ~512 tokens for every non-DeBERTa model.** At batch=1 with bf16, P50 at 512 is typically within 1.2× of P50 at 32. The O(N²) attention cost is dominated by kernel-launch and memory-dispatch overhead until the per-kernel work exceeds the launch cost — which on a 3090 happens around 1024 tokens for dense-attention models (see gte-base: 5 ms @ 512, 8.5 ms @ 1024).

**Fewer layers beats fewer params at batch=1.** bge-micro-v2 (17.4M, 3L/384H) at 1.5 ms P50 is faster than testsavantai-tiny (4.4M, 6L/384H) at ~1.1 ms only marginally, and *faster* than its architectural sibling minilm-L6-H384 (22.7M, 6L/384H) at 2.7 ms. Halving layer count halves the kernel-launch count; halving param count does not. Implication: if a sub-10M filter is wanted on GPU, depth matters more than width or param budget.

**The DistilBERT / BERT-L4 family dominates the 11-67M slot.** bert-L4-H256 (11.2M) at 2.0 ms, bert-L4-H512 (28.8M) at 1.8-2.8 ms, distilbert-base (67.0M) at 2.4 ms. No model outside the DeBERTa family exceeds 3.3 ms P50 at 512 in this size bracket.

**DeBERTa CUDA tax quantified:**

| Model | Params | CUDA P50 @ 512 | vs distilbert-base |
|:---|---:|---:|---:|
| distilbert-base (reference) | 67.0M | 2.44 | 1.0× |
| deberta-v3-xsmall | **70.8M** | 12.49 | **5.1×** |
| deberta-v3-small | 141.9M | 8.40 | 3.4× |
| protectai-deberta-v2 | 184.4M | 14.05 | 5.8× |
| deepset-deberta-injection | 184.4M | 13.50 | 5.5× |

xsmall (70.8M) slower than small (141.9M) is a persistent anomaly; param count is not the driver. Candidate: disentangled relative-position attention dispatches more kernels per layer at xsmall's hidden-size / head-dim ratio than at small's. Needs a torch.profiler trace to confirm (§6 TODO 2).

**testsavantai-base looks like DistilBERT, not DeBERTa.** `src/models.py` tags it `family="deberta"` and the card claims "DeBERTa-v3 base, 86M", but measured params are 67.0M and CUDA latency (2.12 ms @ 512) sits at the DistilBERT-67M profile, not DeBERTa-v3-xsmall-70.8M (12.49 ms). Either the card is wrong or the checkpoint deserializes as DistilBERT. §6 TODO 3.

**MobileBERT is a GPU anti-pattern.** 24.6M params but 17.7–18.5 ms P50 — 7× worse than bge-small (33.4M, 5 ms) and 10× worse than bge-micro (17.4M, 1.7 ms). 24 layers × narrow inverted-bottleneck → 1111 weight tensors (per `Loading weights: 1111/1111`) → kernel-launch cost dominates. Designed for on-device CPU inference, does not translate to server GPU.

**ALBERT is a GPU anti-pattern.** 11.7M params, 5.8–7.0 ms P50 — 3× slower than bert-L4-H256 (11.2M, 2.0 ms) at matched params. ALBERT's cross-layer parameter sharing is a memory-footprint win; compute-wise every layer still runs, just against the same weights. Deep-narrow architecture → kernel-launch overhead per layer.

**Electra-small is unremarkable.** 13.5M, 4.8–6.0 ms P50. Factorized embedding saves params on the embedding matrix, does not reduce the transformer body's compute. No GPU win over similar-size BERT variants.

Three regimes hold, plus one anti-pattern:

1. **Sub-10M dispatch-bound** — GPU speedup modest (2–5×). testsavantai-tiny, bert-L4-H256, bge-micro.
2. **10–50M transitional** — 8–16× GPU speedup.
3. **50M+ GEMM-bound** — 24–50× GPU speedup for non-DeBERTa; 9–30× for DeBERTa.
4. **Architectural anti-patterns** — MobileBERT's 3× and ALBERT's 30× CPU/GPU ratios mask the fact that both are slow everywhere. Per-param cost on CPU is 7–30× worse than the DistilBERT / bge reference at equivalent budget. Avoid.

## 5. Implications for A6 backbone choice

The parent [REPORT.md](../../REPORT.md) §8 "Build Recipe" picks a 66M-class encoder (DistilBERT / ModernBERT-small / DeBERTa-v3-xsmall) on accuracy grounds, subject to P99 ≤ 200 ms CPU and max FPR ≤ 5%. This latency report adds:

1. **If A6 is CPU-deployed at ≤ 512 tokens: 66M DistilBERT Pareto winner holds.** fmops-distilbert @ 83 ms P50 / 512 tok on x86 = 2.4× under budget. 184M DeBERTa is out. ALBERT is out per-param.
2. **If A6 is GPU-deployed at ≤ 512 tokens: every 67M-class distilbert or minilm family fits trivially.** Choose by accuracy and $ alone. Latency is not a constraint.
3. **If A6 is GPU-deployed and needs > 512 tokens: gte-base-en-v1.5 is the new Pareto winner.** 137.4M, 8192 max_pos, 39 ms P50 @ 4096 on a 3090 (1.7× faster than ModernBERT-base / Ettin-150m at matched params). If the accuracy recipe lands on ModernBERT/Ettin, budget GPU cost 1.7×.
4. **Avoid DeBERTa-v3 on GPU** unless accuracy demands it. 5× CUDA overhead vs DistilBERT at equal params means 5× fewer concurrent streams per GPU at equal latency budget.
5. **Avoid MobileBERT and ALBERT universally.** Both lose ≥3× per-param vs any DistilBERT / BGE / BERT-L4 alternative on both CPU and GPU. No deployment path benefits.
6. **Sub-10M "first-gate" filters: CPU > GPU at batch=1.** testsavantai-tiny on CPU at 0.87 ms @ 32 tok beats its own GPU runtime. bge-micro-v2 at 2.4 ms CPU @ 32 is the next step up.

## 6. Unresolved questions (TODO)

Each TODO names the experiment that would resolve it.

1. **Explain the deberta-v3-xsmall / deberta-v3-small CUDA inversion.** xsmall (70.8M) runs 12.5 ms @ 512; small (141.9M) runs 8.4 ms. Param count doesn't explain the ordering. Candidate causes: (a) different disentangled-attention hidden-size splits trigger different kernel implementations, (b) xsmall's shared-embedding scheme forces an extra kernel, (c) autotune picks a worse path at xsmall's shape. Needs `torch.profiler` trace on both models, same input.
2. **Verify testsavantai-base backbone family.** Card claims DeBERTa-v3-base (86M); measured 67.0M + CUDA/CPU latency matches DistilBERT-67M. Check `config.json` in the checkpoint to confirm architecture; fix `src/models.py` family tag if the card is wrong.
3. **Explain gte-base's long-context advantage over ModernBERT.** gte-base-en-v1.5 is 1.7× faster at 4096 tokens than ModernBERT-base / Ettin at nearly identical param count. Candidate: ModernBERT's alternating local/global attention dispatches a global-attention kernel every few layers that does not fuse as cleanly as gte's RoPE-standard-attention. Trace would confirm; practical takeaway holds regardless.
4. **`nomic-ai/nomic-embed-text-v1.5` integration deferred.** Custom modeling.py (a) crashes without explicit `classifier_dropout`, (b) uses `strict=True` on `load_state_dict` so a random-head arch-baseline load fails on missing pooler + classifier weights. Adding it would need a per-model loader branch; not worth it while ModernBERT + Ettin + GTE already cover the long-context cell.
5. **`prajjwal1/bert-{tiny,mini}` tokenizer fix.** `use_fast=False` fallback in `src/bench.py` still routes through `PreTrainedTokenizerFast.__init__` which requires `sentencepiece` / `tiktoken`. Either (a) add `sentencepiece` to `pyproject.toml` and retry, or (b) accept the Google `bert_uncased_L-4_H-{256,512}` replacements and drop the prajjwal repos from `src/models.py`. (b) is cheaper; (a) preserves the published 2L/4.4M and 4L/11M data points.
6. **torch.compile / TensorRT / ONNX-Runtime-CUDA speedup.** Eager PyTorch is a lower bound. Measure `torch.compile(mode="reduce-overhead")` on DistilBERT and on DeBERTa-v3-xsmall; expect ~2× from CUDA Graphs for the dispatch-bound regime, potentially 3–5× on DeBERTa because its kernel-launch overhead is the bottleneck. ORT-CUDA and TensorRT likely collapse the DeBERTa CUDA tax — cheapest path to "DeBERTa at DistilBERT speed".
7. **Batched throughput (batch=4, 16, 64).** At batch=1 GPU is dispatch-bound for small models; batched throughput is what actually matters for multi-tenant serving. Add a `--batch-size` arg to `src/bench.py`, sweep `{1, 4, 16, 64}` × lengths × PI-trained models. Expected finding: the DeBERTa tax shrinks at batch > 16 because kernel-launch cost amortizes.
8. **MobileBERT at batch > 1.** MobileBERT is marketed as a mobile-CPU design; at batch=1 server CPU it runs 15–58 ms but at server GPU it runs 17–18 ms. At batch=4 or 16 the per-sample amortization might collapse the gap. Worth one data point to confirm the design intent.
9. **INT8 / AWQ quantization on CPU.** Parent REPORT.md §9 item 2 lists ONNX INT8 as planned. Add a sister `latency_cpu_int8/` sub-project; expected 2–4× CPU speedup on x86-VNNI. Would close the 184M DeBERTa CPU cliff.
10. **Memory ceiling at batch > 1 on long context.** 3090 has 24 GB; batch=1 @ 4096 fits all three long-context models. batch=16 × 184M × 4096 may not. Add `torch.cuda.max_memory_allocated()` collection once TODO 7 is in; flag which (batch, length) cells OOM.

## Appendix D. Reproducibility

```bash
cd projects/benchmark/benchmark_impl/latency_cuda
bash run.sh          # syncs deps, pre-fetches weights, runs CUDA + CPU pass
```

Outputs: `results/latency_cuda.json`, `results/latency_cpu.json` (schema in [README.md](README.md)). Raw stdout from the 2026-04-20 run: `results/3090_run_2026-04-20.log`; previous run at `results/3090_run_2026-04-16.log`.

Regenerating a partial JSON from a stdout log (when the JSONs are lost but stdout is preserved):

```bash
uv run python scripts/rebuild_json_from_log.py results/3090_run_YYYY-MM-DD.log
```

To rerun a subset:

```bash
uv run python src/bench.py --device cuda --models pi-trained --lengths 32 64 128 256 512
uv run python src/bench.py --device cpu  --models fmops-distilbert,distilbert-base
```
